/pane Architecture in Motion

Pane is the trusted local operating layer for shell behavior, routing, governance, and presentation. It is designed as infrastructure from the start: signed, audited, ACL-protected, rollbackable, and explicit about what is source versus what is live runtime.

Instead of hiding risk behind convenience, Pane makes the system readable. If a change affects trust, execution, or shell integration, it gets treated like infrastructure work, not casual UI work.

Security

Trusted runtime boundaries, protected shell surfaces, and no silent drift from source into execution.

Forensics

Actions leave traces, incidents can be reconstructed, and the system can explain what happened.

Governance

Policy proposes before it applies, rules stay reviewable, and runtime changes stay controlled.

Presentation

Clean hierarchy, sparse surfaces, and a consumer view that helps people work instead of hunt.

How The Stack Fits Source -> Runtime -> Policy

Source

Editable trees such as Hawk source, SWEEP logic, imports, scripts, and rules. This is where systems are shaped, tested, and reviewed before anything becomes live.

Runtime

Published binaries, live imports, shell surfaces, and executable behavior. Runtime is treated as deployed infrastructure, not as a convenient scratchpad.

Policy

Governed metadata and rule promotion through systems like SWEEP and mcp.ini. The system may learn, but it must propose before it applies.

Operating Doctrine Project Rules

Security Before Convenience

If a feature can influence the OS, launch commands, or alter runtime behavior, it is designed as security-sensitive from day one. Trust is structural, not decorative.

Forensics Must Exist

The system should be able to explain what happened after the fact. Logs, traces, and reviewable actions are part of the product, not cleanup work after the product.

Governance Beats Drift

Rules, folder doctrine, publish steps, and runtime boundaries are expected to be explicit. Silent policy mutation and casual runtime edits are treated as design failures.

Presentation Serves Clarity

DOS root structure, menu hierarchy, labels, and visual surfaces exist to keep the system legible. Presentation is part of control because people need to read what the machine is doing.
